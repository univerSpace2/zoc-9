# ZOC9 Product Requirements Document (v1)

- 작성일: 2026-02-28
- 상태: Draft v1 (구현 기준 확정)
- 제품명: ZOC9
- 플랫폼: 모바일 웹(PWA)
- 기본 로케일: `ko-KR`
- 기본 시간대: `Asia/Seoul`

## 1. 제품 목표
ZOC9는 족구 모임 운영자가 경기 규칙에 맞춰 매치를 기록하고, 완료된 기록을 신뢰성 있게 보관/조회할 수 있도록 돕는 모바일 웹 서비스다.

핵심 목표는 다음 3가지다.
1. 모임 현장에서 빠른 기록 입력: 세트 플레이 중 즉시 득점 입력, 서브권/로테이션 자동 처리.
2. 규칙 일관성 보장: 듀스/선승제/세트 시작 서브 규칙을 엔진으로 강제.
3. 운영 신뢰성: 완료 기록 잠금, 관리자 예외 수정, 감사로그 보관.

## 2. 타겟 사용자
- 그룹장/관리자: 모임 생성, 멤버/권한/초대/구장 관리, 완료 처리.
- 일반 멤버: 모임 참여, 매치/세트 기록 조회, 권한 범위 내 기록 입력.
- 심판(선택 역할): 매치 진행 중 점수 기록 보조.

## 3. 성공 지표 (v1)
- 모임 시작 후 첫 매치 생성까지 평균 소요 시간 2분 이내.
- 득점 입력 후 UI 반영 지연 체감 1초 미만.
- 규칙 위반 상태 저장 0건 (서버 검증 기준).
- 완료 기록 수정 이벤트 100% 감사로그 적재.

## 4. 범위 정의
### 4.1 In Scope
- 인증: 이메일/비밀번호 로그인/회원가입, 프로필 수정, 비밀번호 변경.
- 그룹: 그룹 선택, 단일 그룹 자동 입장, 멤버 목록/관리, 권한 기반 액션.
- 모임: 생성/상태 전환(예정→진행중→완료), 상세 조회.
- 매치: 생성, 세트 운영, 득점 로그, 자동 판정, 매치 완료.
- 통계: 모임 기준 승률 순위 및 기본 승/패 요약.
- 열람 전용 모드: 완료 상태에서 일반 수정 차단.
- 관리자 예외 수정: 권한자만 완료 기록 수정 가능 + 감사로그.
- PWA: 홈 화면 추가, 캐시 전략, 절전/복귀 대응, 오프라인 기록 큐.

### 4.2 Out of Scope
- 전체 기능 오프라인 우선 동작.
- 고급 통계 대시보드(장기 추세/복합 지표).
- 네이티브 앱 스토어 배포.
- 결제/정산 자동화.

## 5. 기능 요구사항
### 5.1 인증/계정
- 회원가입 필수 입력: 이메일, 이름, 전화번호.
- 회원가입 선택 입력: 계좌번호.
- 로그인 후 `groups` 진입.
- 내 정보에서 이름/전화번호/계좌번호 수정 및 비밀번호 변경.

### 5.2 그룹/권한
- 그룹 목록 표시, 그룹 1개면 자동 입장.
- 역할: `owner`, `admin`, `member`.
- 하이브리드 권한: 핵심 권한 키만 토글 가능.
- 개인정보 정책(v1): 동일 그룹 멤버 간 전화번호/계좌번호 조회/복사 허용.

### 5.3 초대
- 로그인 전 초대 링크 진입 지원.
- 초대 정보(그룹명/초대자/권한/만료) 노출.
- 로그인 또는 회원가입 후 수락/거절 가능.

### 5.4 모임
- 모임 생성 시 구장/멤버/날짜/시작시간 입력.
- 상태 전환: 예정→진행중→완료.
- 완료된 모임은 기록 입력 화면 비활성(권한자 예외 정책 제외).

### 5.5 매치/세트 기록
- 매치 생성: 팀 구성(2vs2/3vs3/4vs4), 경기 방식(단판/3판2선승/5판3선승), 목표점수(기본15), 듀스(기본on), 벌칙 텍스트.
- 세트 시작: 시작 서브팀 지정(다음 세트는 자동 반대팀 기본값), 초기 서브 포지션 1번 고정.
- 득점 기록: 이벤트 로그(시간 포함) 저장.
- 규칙 엔진: 서브권/로테이션/세트 종료/매치 종료 자동 반영.
- 조기 종료: 필요 승수 충족 시 잔여 세트 `ignored` 처리.

### 5.6 통계
- 모임 기준 멤버 승률 랭킹.
- 멤버별 승/패 요약.
- 완료 데이터만 집계.

### 5.7 완료 기록 열람/수정 정책
- 기본: 완료된 모임/매치/세트는 수정 UI 미노출.
- 예외: `edit_completed_records` 권한자만 수정 가능.
- 예외 수정 시 `audit_logs`에 before/after, actor, timestamp 저장.

## 6. 비기능 요구사항
### 6.1 성능
- 라우트 전환 체감 지연 최소화.
- 목록 조회는 캐시 우선 + 백그라운드 갱신.
- 라이브 기록 화면에서 불필요 리렌더 최소화.

### 6.2 안정성
- 이벤트 단위 멱등 저장(`client_event_id` unique).
- 네트워크 불안정 시 로컬 큐 적재 후 복구 동기화.
- 서버 검증 결과를 정본으로 채택.

### 6.3 보안
- 핵심 테이블 `group_id` 기반 RLS.
- 역할 및 권한 검증은 DB 레이어(RPC/RLS)에서 최종 강제.

### 6.4 관측성
- 완료 수정 감사로그.
- 동기화 실패 이벤트 로깅(추후 Sentry 연동 대상).

## 7. 정보 구조 및 라우팅
- 공개: `/`, `/login`, `/signup`, `/invite/:token`
- 로그인 후 공통: `/groups`, `/profile`
- 그룹 홈 탭: `/g/:groupId/meetings`, `/g/:groupId/match`, `/g/:groupId/members`, `/g/:groupId/more`
- 모임 홈 탭: `/g/:groupId/m/:meetingId/matches`, `/stats`, `/info`
- 라이브 기록: `/g/:groupId/m/:meetingId/match/:matchId/set/:setId/live`

## 8. 데이터 모델 (요약)
주요 테이블: `profiles`, `groups`, `group_members`, `group_permissions`, `invites`, `venues`, `meetings`, `meeting_participants`, `matches`, `match_teams`, `match_players`, `sets`, `set_events`, `notices`, `audit_logs`.

## 9. API/RPC (요약)
- `rpc_accept_invite(invite_token text)`
- `rpc_create_match(payload jsonb)`
- `rpc_start_set(match_id uuid, set_no int, first_serving_team_id uuid)`
- `rpc_record_rally(set_id uuid, scoring_team_id uuid, client_event_id text, occurred_at timestamptz)`
- `rpc_finalize_set(set_id uuid)`
- `rpc_finalize_match(match_id uuid)`
- `rpc_complete_meeting(meeting_id uuid)`
- `rpc_edit_completed_record(payload jsonb)`

## 10. 테스트/수용 기준
### 10.1 단위 테스트
- 2/3/4인 로테이션 순환 검증.
- 서브권 획득 시에만 로테이션 검증.
- 듀스 on/off 세트 종료 판정 검증.
- 선승제 조기 종료와 ignored 세트 처리 검증.

### 10.2 통합 테스트
- 완료수정 권한 검증(거부/허용).
- 완료수정 시 감사로그 생성 검증.
- 초대 상태 전이 검증(대기/수락/만료/거절).

### 10.3 E2E 테스트
- 가입→로그인→그룹 입장.
- 모임 생성→진행중 전환→매치 생성.
- 세트 라이브 기록→매치 종료→읽기 전용 전환.
- 오프라인 기록→복귀 동기화.
- 그룹 홈 매치 탭 바로가기 동작.

## 11. 릴리즈 기준 (Go/No-Go)
다음 항목을 모두 만족해야 v1 배포한다.
1. 핵심 사용자 흐름 E2E 5개 시나리오 통과.
2. 규칙 엔진 단위 테스트 100% 통과.
3. RLS 정책으로 타 그룹 데이터 접근 차단 검증 완료.
4. 완료 수정 감사로그 누락 0건.
5. PWA 설치/복귀/오프라인 큐 동기화 수동 QA 통과.
